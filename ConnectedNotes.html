Aspect Oriented Programming

Core

  • Join point
    A (point)/(the java method) in program execution where advice can be applied
  • Pointcut
    An expression that selects join points where advice should run.
    Pointcuts use expressions based on method names in Java.
  • Advice The (code)/(method) to execute at a join point.
    There are different types of advice:
    • Before: Run before the join point
    • After: Run after the join point
    • Around: Run before and after the join point
  • Aspect A class containing advices and pointcuts related to a cross-cutting concern, like logging, security etc.
  • Weaving The process of applying advice at join points selected by pointcuts.
    It is done by the AOP framework. 
		class Aspect {

			@Pointcut("execution(public * *(..))")
			public void advice1() {} // the pointcut signature

			@Pointcut("within(com.xyz.trading..*)")
			public void advice2() {}

			@Pointcut("advice1() && advice2()") // condition reuse, not methods
			public void advice3() {}
		}

		package com.xyz.trading;

			int methodJoinPoint() {return 0;}

Binary Tree Tree Data Structures

Rooted tree DS in which each node can have maximum 2 child nodes: left and right
Binary Tree can be:
  • Full - every node has either 0 or 2 children
  • Complete - every level, except possibly the last, is completely filled, and all nodes in the last level are as far left as possible
  • Perfect - always Full and Complete
  • Balanced
  • Degenerate

Database normal forms

Terminology:
Table is a Relation.
Elementary key - Can be composite. [Def. 1] Is a key that uniquely determines at most one non-key attribute. [Def. Hugo Kornelis] All other columns depend on the key, and none of these is a cheater FD. Example: {ABC} → XYZ, where {ABC} is an elementary key, than there should not be FDs like B → Y, A → X etc.
Elementary key attribute - an attribute of an elementary key, always single and atomic.
Super Key – a set(superset) of attributes that uniquely identifies a record, but it may include extra attributes that are not necessary for uniqueness.
Candidate Key (or just the key) - [Def. 1] Any set of columns that have a unique combination of values in each row. [Def. 2] A minimal set of attributes that uniquely identifies a record in a Relation. It contains no unnecessary attributes. [Def. Hugo Kornelis] All other columns depend on the key.
Primary Key – a selected candidate key used as the main identifier for records in a Relation.
Non-prime attribute - an attribute that is not a part of any candidate key.
Non-key attribute - an attribute that is not part of the primary key, but may still be part of an alternate (non-primary) candidate key.
Foreign Key – a field in one table that refers to the primary key in another table, establishing a relationship between tables.

Functional Dependency

Determinant → Dependent
"X → Y" means that X determines the value of Y, where X & Y are the attributes of a R.
  • Trivial FD
    A dependent is always a subset of the determinant.
    If X → Y and Y is the subset of X, then it is called trivial functional dependency.
    Examples: A → A, {A,B,C} → {A,B}, {A,B,C} → A, {A,B,C} → {A,B,C}.
  • Non-Trivial FD
    The dependent is strictly not a subset of the determinant.
    If X → Y and Y is not a subset of X, then it is called Non-trivial functional dependency.
  • Multivalued FD
    Entities of the dependent set do not depend on each other.
    A ↠ B and A ↠ C, where is no FD between B & C. Alternate writing A → {B,C}.
  • Transitive FD
    Dependent is indirectly dependent on another determinant.
    If A → B & B → C, then A → C.
  • Elementary FD
    X → A, where
    X is a set of one or more attributes.
    A is a single attribute.
    A does not appear in any smaller dependency with a proper subset of X. Means it is non-decomposable.
    Examples: AB → C, A →/C, B →/C, then AB → C is both fully functional and elementary.
  • Fully FD
    A non-prime attribute depends on the whole candidate key, not only on some portion of it.
    {OrderID,ProductID} → Quantity, where Quantity depends on the whole key.
    StudentID → StudentName.
    AB → CD can be fully FD, but cannot be elementary FD.
  • Partial FD
    A non key attribute depends on a part of the composite key, rather than the whole key.
    X, Y, Z where X and Y are the composite key and Z is a non key attribute. Then X → Z and Y → Z are the partial FD.

Object-Oriented Programming

Encapsulation, Abstraction, Inheritance, Polymorphism.
Relationships:
  • Inheritance uses is-a relationship
  • Association is a relationship between two independent classes where objects of one class use or interact with objects of another class.
    It is a "using" relationship that can be one-to-one, one-to-many, or many-to-many, and both objects have their own lifespans.
    Association defines structural relationship between classes of objects that allows one object instance to cause another to perform an action on its behalf.
    Key characteristics of association: Independent objects, "Using" relationship, Unidirectional or bidirectional.
    Special forms(extension) of Association:
    • Composition uses has-a relationship. Related objects are the part of the container, so with destruction of a container they will be destroyed too. It is a Strong association
    • Aggregation uses has-a relationship. Container only hold the references of contained objects. Destruction of a container does not affect them. It is a Weak association and one-way relationship.

Privacy by Design

  1. Principle #1: Proactive, Not Reactive; Preventive, Not Remedial
    No amount of privacy measures post factum can really keep a fundamentally non-private technology or system from enabling the misuse of personal data.
  2. Principle #2: User Privacy as the Default Setting
    User should already have the highest privacy by default.
  3. Principle #3: Privacy Embedded Into Design
    Start the privacy and security conversation when planning your system. You will find that it’s quite difficult and inefficient to try and add these features later in an already-built technology.
  4. Principle #4: Full Functionality—Positive-Sum, Not Zero-Sum
    Privacy doesn’t have to come at the expense of product benefits. If done properly, privacy-by-design principles could help you create a better, more user-friendly product.
  5. Principle #5: End-to-End Security—Full Lifecycle Protection
    The lifecycle of personal data starts at the point of collection and ends when it’s finally destroyed or deleted after you no longer need it. Consumer data should be protected right from the start and up until it’s been deleted.
  6. Principle #6: Visibility and Transparency—Keep it Open
    You should have a well-documented privacy policy and engage in transparent communication about the data processing and privacy-protecting activities you undertake.
  7. Principle #7: Respect for User Privacy—Keep It User-Centric
    The system should have intuitive tools that enable users to manage their data.
    User should have the interface to change their consents.

Red Black Tree Tree Data Structures

TreeMap is based on RBT.
RBT is a self-balancing Binary Search Tree(BST) data structure.

Requirements:
From BST:
  1. Each key is greater than any key to the node's left subtree AND less than in its right subtree
From RBT:
  1. Every node is either red or black
  2. All NIL nodes are considered black
  3. A red node does not have a red child
  4. Every path from a given node to any of its descendant NIL nodes goes through the same number of black nodes
  5. (Conclusion) If a node N has exactly one child, the child must be red.
    Because if it were black, its NIL descendants would sit at a different black depth than N's NIL child, violating requirement 4

RBT is a self-balancing, that means it must rearrange itself on add and delete operations.
The inner self-balancing operation are: left and right shift (means switching parent and child), switch color.

Space Complexity: O(n) The memory space that algorithm uses in the process of work, depending on amount of input data.
Time complexity (Worst cases (and amortized)):
Search, Insert and Delete: O(log2 n)

REST (Representational State Transfer)

Is a software architectural style that was created to describe the design and guide the development of the architecture for the WWW.
REST defines a set of constraints for how the architecture of a distributed, Internet-scale hypermedia system, such as the Web, should behave.
The REST architectural style defines 6 guiding constraints. ⚓SUL3C
When these constraints are applied to the system architecture, it gains desirable NFR properties:
performance scalability simplicity modifiability visibility portability reliability.
  1. Client/Server – clients are separated from servers by a well-defined interface
  2. Stateless – a specific client does not consume server storage when the client is "at rest", that also means sessionless
  3. Cache – responses indicate their own cacheability. Servers specify cacheability of responses through HTTP headers like Cache-Control, Expires and Etag.
  4. Uniform interface is fundamental to the design of any RESTful system. It simplifies and decouples the architecture, which enables each part to evolve independently.
    1. Resource identification in requests - resources are identified by URI.
      The resources themselves are conceptually separate from the representations(HTML, XML, JSON) that are returned to the client. GET /users/123
    2. Resource manipulation through representations - When a client holds a representation of a resource, including any metadata attached, it has enough information to modify or delete the resource's state. PUT /users/123
    3. Self-descriptive messages - Each message includes enough information to describe how to process the message. (media type)
    4. HATEOAS - a REST client should be able to use server-provided links dynamically to discover all the available resources it needs.
      The server responds with text that includes hyperlinks to other resources that are currently available
  5. Layered system – a client cannot ordinarily tell whether it is connected directly to the end server, or to an intermediary along the way
    Web app layers example
    ClientWeb browser/Mobile App
    IntermediaryCDN, reverse proxy for load balancing & caching
    ServerApplication server & Business logic
    Database
  6. Code on demand (optional) – servers are able to temporarily extend or customize the functionality of a client by transferring logic to the client that can be executed within a standard virtual machine.
    Example: Javascript
Richardson Maturity Model
  1. Level 0: The Swamp of POX ⚓ 1 URI & 1 method
    Is not RESTful. It is a Web API with a single URI accepting all the range of operations supported by the service.
    Resources are not well-defined.
    Example: /bookingService POST retrieve destinations/hotels/rooms; add/cancel a reservation; etc.
  2. Level 1: Resources ⚓ n URI, but still 1 method
    Still not yet RESTful. Introduces resources and allows requests for individual URIs (still all typically POST) for separate actions instead of exposing one universal endpoint (API).
    The API resources are still generalized, but it is possible to identify the scope of each one. Example:
    /bookingDestinations POST retrieve destinations
    /bookingReservations POST add/cancel reservations
    /bookingRooms POST add/cancel special requests to a reservation
    /bookingFeedback POST leave feedback
  3. Level 2: HTTP verbs (HTTP request methods) ⚓ n URI & n methods
    It is RESTful. The system starts making use of HTTP Verbs. This allows for further specialization of the resource and thus narrows down the functionality of each individual operation within the service. The principal separation at this level consists in splitting a given resource into two – one request for obtaining data only (GET), the other for modifying the data (POST). Further granularization is also possible.
    Example:
    /destinations GET retrieve destinations
    /reservations GET get reservations according to certain criteria
    /reservations POST add/cancel reservations
    /rooms POST request room extras
    /rooms DELETE remove room extras
  4. Level 3: Hypermedia controls
    It is RESTful. The last level introduces the hypermedia representation. Also called HATEOAS. A client interacts with a network application whose application servers provide information dynamically through hypermedia. A REST client needs little to no prior knowledge about how to interact with an application or server beyond a generic understanding of hypermedia.
    The restrictions imposed by HATEOAS decouple client and server. This enables server functionality to evolve independently.
    Example:
    GET /accounts/12345 HTTP/1.1

SOLID

  • Single responsibility principle
    There should never be more than one reason for a class to be changed. Means class code can be changed, and it should be only for 1 type of activity.
    In other words, every class should have only one responsibility.
    "God-object" is an example of the antipattern of this.
  • Open–closed principle
    Software entities should be open for extension, but closed for modification.
    That is, such an entity can allow its behaviour to be extended without modifying its source code.
    In other words, from the very beginning, the code should be written in a way it should not be changed in the future, in case the functionality is needed to be extended.
    Pros: Less regression testing. less errors
    Java example
  • Liskov substitution principle
    Functions/methods that use pointers or references to base classes must be able to use objects of derived classes without knowing it.
    Java example
  • Interface segregation principle
    No object/code should be forced to depend on methods it does not use.
    Java example
  • Dependency inversion principle
    1. High-level modules should not depend/import on low-level modules, but both should depend on abstractions.
    2. Abstractions should not depend on details. Details (concrete implementations) should depend on abstractions.
    Modules should be loosely coupled.
    The 2nd part means that interfaces and abstract classes should not reference any implementations, but should only reference interfaces and abstract classes.
    Final Notes:
    - Implementations classes should not extend other implementations, only interfaces and abstract classes
    - Method parameter's should not reference implementations, only interfaces and abstract classes
    Java example

Transaction Isolation

Read Phenomena/Issues

  1. Dirty reads

    A dirty read (aka uncommitted dependency) occurs when a transaction retrieves a row that has been updated by another transaction that is not yet committed.

  2. Non-repeatable reads

    A non-repeatable read occurs when a transaction retrieves a row twice and that row is updated by another transaction that is committed in between.

  3. Phantom reads

    A phantom read occurs when a transaction retrieves a set of rows twice and new rows are inserted into or removed from that set by another transaction that is committed in between.

Isolation levels (SQL92 standard)

  1. Read uncommitted

    This is the lowest isolation level. In this level, dirty reads are allowed, so one transaction may see not-yet-committed changes made by other transactions.

  2. Read committed

    In this isolation level, a lock-based concurrency control DBMS implementation keeps write locks (acquired on selected data) until the end of the transaction, but read locks are released as soon as the SELECT operation is performed (so the non-repeatable reads phenomenon can occur in this isolation level).
    Simpler: read committed is an isolation level that guarantees that any data read is committed at the moment it is read. It simply restricts the reader from seeing any intermediate, uncommitted, 'dirty' read.

  3. Repeatable reads

    A lock-based concurrency control DBMS implementation keeps read and write locks (acquired on selected data) until the end of the transaction. Range-locks are not managed, so phantom reads can occur.
    Write skew is possible in some systems. Write skew is a phenomenon where two writes are allowed to the same column(s) in a table by two different writers (who have previously read the columns they are updating), resulting in the column having data that is a mix of the two transactions.

  4. Serializable

    This is the highest isolation level.
    With a lock-based concurrency control DBMS implementation, serializability requires read and write locks (acquired on selected data) to be released at the end of the transaction. Also, range-locks must be acquired when a SELECT query uses a ranged WHERE clause, especially to avoid the phantom reads phenomenon.
